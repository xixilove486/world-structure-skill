> [!IMPORTANT]
> **世界结构.skill** 不是“解释万物”的终极神谕。  
> 它更像一册可以反复翻阅的**结构诊断手记**：  
> 当团队、关系、项目、组织或公共议题开始失衡时，陪你从**锚点、约束、角色、推力链、扰动、熵增与演化阶段**出发，慢慢判断一个系统如何形成、维持、失衡、转折、更新与崩解。

<a id="why"></a>
## 为什么需要这个 skill？

人们分析问题时，往往很容易掉进几个常见误区。  
这些误区并不总是因为不够聪明，而常常只是因为我们太容易被最显眼的部分带走。

### 1. 只看表面现象
看到争吵，就以为问题是情绪失控。  
看到停滞，就以为问题是没人努力。  
看到崩坏，就以为问题是某个人出了问题。  

但很多时候，表面现象只是结果，不是根源。

### 2. 只从“人性”出发解释一切
一遇到复杂问题，就急着归结为自私、欲望、背叛、嫉妒、权力。  
这些解释有时并不完全错，但它们常常过早，而且过浅。  

因为很多问题不是先由“人性恶”引起的，  
而是由**结构失衡、角色错位、推力链断裂、阶段错判、边界失效**逐步推出来的。

### 3. 忽略结构的作用
人们很擅长讨论动机，却不擅长讨论系统。  
很擅长判断谁对谁错，却不擅长判断：

- 这个系统靠什么维持？
- 它现在被什么约束？
- 什么在慢慢让它漂移、空心化、僵化或碎裂？
- 它到底处于形成、扩张、瓶颈，还是已经进入衰退？

而这，恰恰决定了问题能不能被真正看清。

<a id="what"></a>
## 这个 skill 能做什么？

**世界结构.skill** 不是一个“人格判断器”，  
也不是一个“解释万物的万能理论”。

它是一套用于分析复杂系统的结构工具。

它试图帮助你：

- 看清一个团队为什么推进不动
- 看清一段关系为什么没有崩，却越来越空
- 看清一个项目为什么越忙越失真
- 看清一个组织为什么从活力走向僵化
- 看清一个制度、社群或共同体为什么会漂移、分裂、空心化，或者重新获得生机

它最核心的能力，不是给出一个漂亮结论，  
而是把复杂问题拆回几个更扎实、也更值得先问的问题：

- 它的锚点是什么？
- 它的约束是什么？
- 谁在维持，谁在推动，谁在阻塞？
- 它的推力链从哪里来？
- 它正被什么扰动？
- 它处于哪个阶段？
- 如果只能先修一处，最该修哪里？

<a id="gain"></a>
## 它会带来什么提升？

这个 skill 想提供的，不只是“分析工具”，  
更是一种更稳、更不容易被表象带跑的观察方式。

它会帮助你逐步获得几种能力：

### 1. 从表象退到结构
不急着被情绪、戏剧性冲突和局部事件牵着走，  
而是先问：**背后的结构到底是什么。**

### 2. 从责备退到诊断
不急着把问题归到某个人头上，  
而是先判断：**这到底是个体问题、角色问题，还是系统问题。**

### 3. 从静态退到演化
不把一个对象看成固定状态，  
而是看成一个正在形成、扩张、失衡、转折或更新的过程。

### 4. 从零散解释退到整体判断
不是只会说“这里有问题”，  
而是能逐步形成对系统整体状态的判断：  
它为什么会这样，它接下来会往哪边走，它最该先修哪里。

换句话说，  
这个 skill 想帮助你获得的，不只是一个答案，  
而是一种更不容易被表面现象欺骗的世界理解方式。

<a id="love"></a>
## 关于“爱”的边界

> [!IMPORTANT]
> ### 爱不是用来替代理论解释的，而是用来标记理论解释的边界。
>
> 在这个模型里，**爱不是一个被随手调用的抒情词**。  
> 它是一个**中性的、具有超越性的概念**：  
> 不是用来跳过结构，也不是用来跳过心理，  
> 而是用来提醒我们，理论已经尽力之后，仍然剩下的那部分行动余量，也许值得被谨慎地留给爱。

<p align="center"><sub>先尽可能完成结构解释，再把无法压平的余量，谨慎地留给爱。</sub></p>

也就是说：

- 能被结构解释的，先别急着抬到爱
- 能被心理机制解释的，也别急着抬到爱
- 只有在结构与心理都足够强，却仍然剩下无法压平的行动余量时，才承认有超越性成分

这不是在贬低爱，  
恰恰相反，这是一种克制的尊重。

因为如果一切都可以随手归因于爱，  
那爱就不再是边界之外的东西，  
而会被滥用成一个偷懒的解释词。

**世界结构.skill** 的立场是：

- 先尽可能完成结构解释
- 再尽可能完成心理解释
- 最后，把无法被压平的那部分余量，谨慎地留给超越性

模型不是为了取消爱。  
模型是为了避免爱被滥用，也为了避免真正的爱被浪费。

## 安装方式

这个 skill 现在支持三种安装方式：

- **Codex 安装（推荐）**：最方便，直接从 GitHub 子路径安装
- **个人安装**：对你所有项目生效
- **项目安装**：只在某一个仓库内共享使用

### Codex 安装（推荐）

#### 方式 1：按 GitHub URL 安装
```powershell
py -3 "$HOME/.codex/skills/.system/skill-installer/scripts/install-skill-from-github.py" --url https://github.com/xixilove486/world-structure-skill/tree/main/skills/world-structure
```

#### 方式 2：按仓库和路径安装
```powershell
py -3 "$HOME/.codex/skills/.system/skill-installer/scripts/install-skill-from-github.py" --repo xixilove486/world-structure-skill --path skills/world-structure
```

安装后重启 Codex，即可识别新 skill。  
如果你只是想最快开始使用，优先选这一种。

当前内置的入口包括：

| 接口 | 入口文件 | 说明 |

| Claude Code | `CLAUDE.md` | 作为 Claude Code 的项目级记忆入口 |

<a id="axes"></a>
## 核心分析主轴

这个 skill 默认从七个方向看系统：

| 维度 | 它在问什么 |
|---|---|
| **锚点** | 系统围绕什么在维持、追求或漂移？ |
| **约束** | 什么在限制它？ |
| **角色与层级** | 谁在维持、谁在推动、谁在连接、谁在阻塞？ |
| **推力链** | 资源、信任、记忆、合法性与情感投入从哪里来？什么在持续把系统往前推、往回拉，或让它变薄、断裂？ |
| **扰动 / 漂移 / 熵增** | 什么在让系统失真、漂移、僵化、分裂或空心化？ |
| **阶段** | 它正处于形成、扩张、瓶颈、转型、稳定、僵化、碎裂还是更新尝试？ |
| **干预优先级** | 先修哪里最值？ |

<a id="principles"></a>
## 设计原则

| 原则 | 含义 |
|---|---|
| **结构优先于责备** | 先看系统，再谈责任 |
| **高成本信号优先于宣称** | 先看行动，再听口号 |
| **约束优先于建议** | 先看承载，再谈改进 |
| **不确定性必须显式保留** | 不把模型当神谕 |

<a id="good-output"></a>
## 一个好输出长什么样？

```text
结构判断
核心矛盾
主要风险方向
最佳干预优先级
不确定性 / 备选解释
```

简洁、清醒、不武器化。  
这比“宏大、激烈、像在审判世界”的输出更有价值。  
一个好的输出更像一页可靠的诊断手记，而不是一份居高临下的判词。

<a id="ideal"></a>
## 理想与现实

> [!NOTE]
> 本模型不预设任何宏大理想必然失败，也不预设任何终极目标会自动实现。
>
> 它关心的是：任何理想一旦进入现实组织与历史过程，就会遭遇异化、熵增、锚点漂移与反馈失真。模型的任务不是取消理想，而是帮助识别理想如何在现实中被损耗、篡改，或在何种条件下被保存。
>
> 如果一个理想不能承受对其现实实现机制的分析，那么真正威胁它的，不是模型，而是它自身的脆弱。
>
> 因此，世界模型不应被表述为对理想的否定，而应被表述为对理想实现条件的严厉审计。
