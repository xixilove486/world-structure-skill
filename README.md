## 为什么需要这个 skill？

人们分析问题时，往往很容易掉进三个常见误区：

### 1. 只看表面现象
看到争吵，就以为问题是情绪失控。  
看到停滞，就以为问题是没人努力。  
看到崩坏，就以为问题是某个人出了问题。

但很多时候，表面现象只是结果，不是根源。

### 2. 只从“人性”出发解释一切
一遇到复杂问题，就急着归结为自私、欲望、权力、背叛、嫉妒。  
这些解释有时并不完全错，但它们常常过早，而且过浅。

因为很多问题不是先由“人性恶”引起的，  
而是由 **结构失衡、角色错位、支持链断裂、阶段错判、边界失效** 逐步推出来的。

### 3. 忽略结构的作用
人们很擅长讨论动机，却不擅长讨论系统。  
很擅长判断谁对谁错，却不擅长判断：

- 这个系统靠什么维持？
- 它现在被什么约束？
- 什么在慢慢让它漂移、空心化、僵化或碎裂？
- 它到底处于形成、扩张、瓶颈，还是已经进入衰退？

而这，恰恰决定了问题能不能被真正看清。

## 这个 skill 能做什么？

**世界结构.skill** 不是一个“人格判断器”，  
也不是一个“解释万物的万能理论”。

它是一套用于分析复杂系统的结构工具。

它试图帮助你：

- 看清一个团队为什么推进不动
- 看清一段关系为什么没有崩，却越来越空
- 看清一个项目为什么越忙越失真
- 看清一个组织为什么从活力走向僵化
- 看清一个制度、社群或共同体为什么会漂移、分裂、空心化，或者重新获得生机

它最核心的能力，不是给出一个漂亮结论，  
而是把复杂问题拆回几个更扎实的问题：

- 它的锚点是什么？
- 它的约束是什么？
- 谁在维持，谁在推动，谁在阻塞？
- 它靠什么获得资源、认同、记忆与支持？
- 它正被什么扰动？
- 它处于哪个阶段？
- 如果只能先修一处，最该修哪里？

## 它会带来什么提升？

这个 skill 想提供的，不只是“分析工具”，  
更是一种更稳的观察方式。

它会帮助你逐步获得几种能力：

### 1. 从表象退到结构
不急着被情绪、戏剧性冲突和局部事件牵着走，  
而是先问：**背后的结构到底是什么。**

### 2. 从责备退到诊断
不急着把问题归到某个人头上，  
而是先判断：**这到底是个体问题、角色问题，还是系统问题。**

### 3. 从静态退到演化
不把一个对象看成固定状态，  
而是看成一个正在形成、扩张、失衡、转折或更新的过程。

### 4. 从零散解释退到整体判断
不是只会说“这里有问题”，  
而是能逐步形成对系统整体状态的判断：
它为什么会这样，它接下来会往哪边走，它最该先修哪里。

换句话说，  
这个 skill 想帮助你获得的，不只是一个答案，  
而是一种更不容易被表面现象欺骗的世界理解方式。

## 关于“爱”的边界

> [!NOTE]
> **爱不是用来替代理论解释的，而是用来标记理论解释的边界。**

也就是说：

- 能被结构解释的，先别急着抬到爱  
- 能被心理机制解释的，也别急着抬到爱  
- 只有在结构与心理都足够强，却仍然剩下无法压平的行动余量时，才承认有超越性成分

这不是在贬低爱，  
恰恰相反，这是一种克制的尊重。

因为如果一切都可以随手归因于爱，  
那爱就不再是边界之外的东西，  
而会被滥用成一个偷懒的解释词。

**世界结构.skill** 的立场是：

- 先尽可能完成结构解释
- 再尽可能完成心理解释
- 最后，把无法被压平的那部分余量，谨慎地留给超越性

模型不是为了取消爱。  
模型是为了避免爱被滥用，也为了避免真正的爱被浪费。

> - 给人贴“本质标签”的工具
> - 用来攻击、羞辱或武器化分析对象的语言系统
> - “我懂你的一切”的人格审判器

## 它能做什么？

这个 skill 主要适合：

- 团队分析
- 关系分析
- 项目诊断
- 组织与制度分析
- 社群与复杂处境分析
- 反复冲突、长期停滞、边界模糊、暗中漂移的系统判断

它最擅长输出五种东西：

1. **Structural Judgment**  
   当前系统发生了什么

2. **Core Contradiction**  
   最深的结构矛盾是什么

3. **Main Risk Direction**  
   如果不改，最可能往哪里坏

4. **Best Intervention Priority**  
   先修哪里最值

5. **Uncertainty / Alternative Readings**  
   还存在哪些不确定项

## Install

This skill can be installed either:

- per-user: available across your repos
- per-repo: shared inside one project

Codex-style skill loaders typically discover skills from standard skill folders and load them progressively, so the full long-form reference is only read when the skill is actually used.

### Per-user install

#### macOS / Linux
```bash
mkdir -p "$HOME/.agents/skills" && \
(test -d "$HOME/.agents/skills/world-structure-skill/.git" && \
git -C "$HOME/.agents/skills/world-structure-skill" pull --ff-only || \
git clone https://github.com/xixilove486/world-structure-skill.git "$HOME/.agents/skills/world-structure-skill")
```

#### Windows PowerShell
```powershell
New-Item -ItemType Directory -Force "$HOME/.agents/skills" | Out-Null
if (Test-Path "$HOME/.agents/skills/world-structure-skill/.git") {
  git -C "$HOME/.agents/skills/world-structure-skill" pull --ff-only
} else {
  git clone https://github.com/xixilove486/world-structure-skill.git "$HOME/.agents/skills/world-structure-skill"
}
```

## 核心分析主轴

这个 skill 默认从七个方向看系统：

| 维度 | 问题 |
|---|---|
| **Anchor** | 系统围绕什么在维持、追求或漂移？ |
| **Constraints** | 什么在限制它？ |
| **Roles & Layers** | 谁在维持、谁在推动、谁在连接、谁在阻塞？ |
| **Support Chains** | 资源、信任、记忆、合法性与情感投入从哪里来？ |
| **Disturbance / Drift / Entropy** | 什么在让系统失真、漂移、僵化、分裂或空心化？ |
| **Stage** | 它正处于形成、扩张、瓶颈、转型、稳定、僵化、碎裂还是更新尝试？ |
| **Intervention Priority** | 先修哪里最值？ |

## 设计原则

<div align="center">

| 原则 | 含义 |
|---|---|
| **结构优先于责备** | 先看系统，再谈责任 |
| **高成本信号优先于宣称** | 先看行动，再听口号 |
| **约束优先于建议** | 先看承载，再谈改进 |
| **不确定性必须显式保留** | 不把模型当神谕 |

</div>

## 一个好输出长什么样？

```text
Structural Judgment
Core Contradiction
Main Risk Direction
Best Intervention Priority
Uncertainty / Alternative Readings
```

简洁、清醒、不武器化。  
这比“宏大、激烈、像在审判世界”的输出更有价值。
